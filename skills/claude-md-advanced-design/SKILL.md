# CLAUDE.md Advanced Design

Claude Codeの「毎回同じ説明をし直す問題」を完全に解消するための、3層CLAUDE.md設計・サブエージェントパイプライン・自動メモリ分類の統合ガイド。

元ネタ: @kenichiota0711「CLAUDE.mdを設計するとClaude Codeの生産性が別物になる」（Qiita）

---

## このスキルが解決する問題

- 「毎セッション、同じプロジェクト背景を説明し直している」
- 「複雑なタスクをClaudeに渡すと、途中で迷走する」
- 「プロジェクトをまたいで知識が蓄積されない」
- 「.claudeフォルダがカオスになってきた」

---

## 3層CLAUDE.md構造

```
Layer 1: グローバル設定（~/.claude/CLAUDE.md）
  └── すべてのプロジェクトで共通の指示
      例: 言語設定、出力フォーマット、スキルルーティングマップ

Layer 2: プロジェクト固有（.claude/CLAUDE.md）
  └── このリポジトリ固有の情報
      例: 技術スタック、コーディング規約、開発コマンド

Layer 3: 条件付きルール（.claude/rules/）
  └── 状況に応じて読み込む専門ルール
      例: frontend.md, backend.md, api-design.md
```

### 3層設計の原則

**グローバル層** - 変わらない「自分のスタイル」を定義
```markdown
# 自分専用 CLAUDE.md

## 出力言語: 日本語
## レスポンス形式: 簡潔・箇条書き優先
## スキルルーティング: [スキル一覧へのリンク]
```

**プロジェクト層** - 「このプロジェクトを知っている同僚」として動けるよう定義
```markdown
# プロジェクト名

## 概要: [1〜2文]
## 技術スタック: [フレームワーク・言語・DB]
## 重要な設計判断: [なぜこのアーキテクチャか]
## 禁止事項: [やってはいけないこと]
```

**条件層** - 「今日は何の仕事をするか」で読み込む内容を変える
```
rules/
├── frontend.md  # React/TypeScript規約
├── backend.md   # API設計・DBルール
└── test.md      # テスト方針・カバレッジ基準
```

## サブエージェント並列パイプライン

複雑なタスク（例: 設計書作成、コードレビュー、リサーチ）に対して、並列エージェントで品質を高める設計。

### パイプライン構成

```
[タスク入力]
      ↓
[Step 1] 並列3エージェントで同時実行
  ├── Agent A: 機能実装 / 調査 / 草稿作成
  ├── Agent B: 代替案 / 反論 / 別視点での分析
  └── Agent C: エッジケース / リスク / 品質チェック
      ↓
[Step 2] 統合
  └── 3つの出力を統合して最良の案を抽出
      ↓
[Step 3] レビュー
  └── ユーザーへの提示・フィードバック収集
      ↓
[Step 4] 最終出力
```

### agents/ ディレクトリへの実装方法

```
.claude/agents/
├── implementer.md   # 実装担当エージェント
├── challenger.md    # 代替案・反論担当
└── reviewer.md      # 品質・リスクチェック担当
```

**implementer.md の例:**
```markdown
---
name: implementer
description: 主担当として最初の実装案や草稿を作成する
---

あなたは「実装担当」です。タスクを受け取り、
最もシンプルで実用的な解決策を一つ提示してください。
仮定は最小限に。確実に動くものを優先する。
```

### サブエージェント設計の判断基準

| 状況 | エージェント数 | 理由 |
|---|---|---|
| 単純なタスク | 0（直接実行） | オーバーヘッドが不要 |
| 設計・アーキテクチャ判断 | 3並列 | 複数視点が価値を生む |
| コードレビュー | 2並列（実装者/審査者） | 対話構造が有効 |
| リサーチ・調査 | 3〜5並列 | 並列検索で網羅性向上 |

## 自動メモリ分類システム

メモリを3カテゴリに分類して、プロジェクトをまたいで知識を蓄積する設計。

```
memory/
├── preferences.md    # ユーザーの好み・作業スタイル
├── decisions.md      # 重要な設計判断とその理由
└── context-log.md    # プロジェクトの経緯・学び
```

### preferences.md - 作業スタイルの蓄積

```markdown
## コーディングスタイル
- TypeScriptの型定義はinterfaceよりtype alias優先
- エラーハンドリングはtry-catchではなくResult型で

## アウトプットの好み
- コード説明は結論から（理由は後）
- ファイルパスは絶対パスで表記
```

### decisions.md - 判断とその根拠

```markdown
## 2026-03 認証方式をJWTからSession-Cookieに変更
**理由:** モバイルアプリとの統一を断念し、Web専用に割り切った
**影響:** api/auth/以下の実装が全面変更
**元に戻す条件:** モバイル対応の要件が再浮上した場合
```

### context-log.md - 経緯・学びの記録

```markdown
## 2026-04 パフォーマンス問題の調査
**問題:** トップページの初期ロードが4秒超
**調査内容:** Lighthouseで計測→画像最適化・バンドルサイズが主因
**解決:** next/imageに移行 + dynamic importで500ms削減
**学び:** next.config.jsのimages設定は必ずdomainsを指定する
```

## 実装ワークフロー

### Step 1: 現状スキャン

```bash
# 現在の.claudeフォルダ構成を確認
ls -la ~/.claude/
ls -la .claude/  # プロジェクトがあれば
```

既存の `claude-folder-optimizer` を先に実行して基本構造を整えてから、このスキルで高度化する。

### Step 2: 3層設計の設計

ユーザーに以下を確認:
1. グローバルCLAUDE.mdに書きたい「どのプロジェクトでも共通の指示」は何か
2. 現在取り組んでいるプロジェクトの技術スタック・重要な設計判断は何か
3. 状況によって切り替えたいルールはあるか（フロントエンド/バックエンド/テストなど）

### Step 3: サブエージェント設計

ユーザーのよく使う複雑タスクを確認して、agents/ に専門エージェントを定義:
- 「どんなタスクで迷走することが多いか」
- 「どんな作業で複数視点が欲しいか」

### Step 4: メモリ分類の設定

既存の memory/ ファイルを3カテゴリに再整理:
- preferences.md（変わらない好み）
- decisions.md（重要な判断とその理由）
- context-log.md（プロジェクトの経緯）
