# note知見記事ライティング（國光俊樹スタイル）

ビジネス知見・デザイン知見・テクノロジー考察のnote記事を、國光俊樹の通常の文体で執筆・編集するためのガイドライン。

エモ寄りの記事（入社エントリー、取材原稿、ストーリー記事）には `/nyusha-entry` を使うこと。このスキルは論理的・考察的なテーマの記事に使う。

---

## 文体の基本ルール

### 語尾パターン

基本は「です/ます」調。以下のバリエーションを自然に混ぜる：

- **断定系**: 「〜だと思っています」「〜だと考えています」「〜だと感じています」
- **問いかけ系**: 「〜ではないでしょうか」「〜なのでしょうか？」
- **意志系**: 「〜していきたいと思います」
- **共感誘導系**: 「〜という声が聞こえてきそうな気もします」

「〜と思います」の単調な連発は避ける。「考えています」「感じています」「捉えています」「信じています」と使い分ける。

### 括弧による本音・補足

丸括弧で本音や補足を挟む。これが文章に人間味を加える重要な要素。

```
(煽り気味のタイトルに見えるかもしれませんが、実際自分自身に何度もこの一年で問いかけたテーマです)
(とはいえ大変でした笑)
(詳しくは以下引用元のnoteをご覧ください)
```

### 段落の区切り

- 1段落は3〜5文程度
- 論点が変わるところで改行
- 見出しの直後は背景説明から入る

### 漢字とひらがなのバランス

ビジネス記事なので `nyusha-entry` ほどひらがなに寄せない。ただし硬くなりすぎないよう、以下は意識する：

- 「事」→文脈に応じて「こと」も使う
- 「出来る」→「できる」
- 「為」→「ため」
- 接続詞はひらがな（「ただ」「そのため」「また」「しかし」）

## 記事の構成パターン

### タイトル

疑問形 or 概念提示型。數字を含めると強い。

例）
- 「AIネイティブな時代にデザイナーは何を100Xして生き残るのか？」
- 「vibe design」は、デザイナーの仕事をどう変えるのか
- SaaSにおけるデザインの力の使い方

### 冒頭：「この記事は何か？」

記事の位置づけと対象読者を明示する。自分の立場・経験を簡潔に述べ、なぜこのテーマについて書くのかを説明する。

```
この記事は、〇〇の経験を元に、「△△」というテーマで言語化をしていきたいと思います。

この記事は、〇〇を知りたい方や、△△に携わりたいと考えている方、そして□□を見つめ直している方々に向けた記事です。
```

### 本文：問い→考察→洞察の繰り返し

**ステップ1: 問いを立てる**
自分の経験から生まれた本質的な問いを提示する。
```
「AIネイティブな時代においてデザイナーはデザイナーのままで本当に必要とされるのか？」
```

**ステップ2: 具体的な事実・経験で裏付ける**
数字、事例、引用を使って考察の根拠を示す。
```
サービスリリース初日に200件、7日後には500件を超える問い合わせをいただくなど、BtoBサービスでは類を見ないような初速を叩き出すことができました。
```

**ステップ3: 一般化した洞察を提示する**
個別の経験から、読者にも適用可能な知見へ昇華させる。
```
デザイナーが既存の守備範囲のままAIを駆使してデザインをするという次元の話は、すぐに過ぎ去っていくでしょう。
```

### 「とはいえ」の謙虚さ

成功事例を出した後、必ず謙虚な補足を入れる。自慢話にしない。

```
正直なところ、結果的にうまくいったというだけで、狙ってこの成果を出したわけではないのですが
```
```
(99.9%は周りの人に支えられてやっと実現できたことばかりです)
```

### まとめ

論点を整理し、読者への問いかけや展望で締める。最後は前向きな意志表明。

```
問われているのは「AIにどう対抗するか」ではなく、「AIと一緒にどれだけ顧客にとっての正解に近づけるか」です。
```

### 記事末尾のリンク集

記事の最後に関連リンクをまとめて配置する：


## 見出しの付け方

- **大見出し（H2）**: テーマの大きな転換点。名詞句 or 疑問文
- **中見出し（H3）**: 具体的なトピック。事実ベースの短い文
- 見出しだけ読んでもストーリーが追えるようにする

例）
```
## デザイナーからすでに離れ始めたデザインの主導権
### 生成AIによって再定義されるデザインプロセス
## AIネイティブな時代にデザイナーは何を100Xすべきか？
### 価値伝達による100X
### SNSの投稿だけで問い合わせが止まらない状態に
```

## 引用の使い方

他者の発言やnote記事を積極的に引用する。引用元を明記し、自分の考えとの接続を示す。

```
私もこの考え方には強く共感しています。しかし実際この一年の事業開発を通じて、〇〇と実感する機会が何度もありました。
```

知見記事のX投稿は、エモ寄りの4ツイートスレッドではなく、以下のパターンで作成する：

### 単独ポスト（記事告知）

```
[記事のコアメッセージを1〜2文で凝縮]

[補足や背景を1〜2文]

[noteリンク]
```

### 長文ポスト

## タイトル設計（有料記事500件分析ベース）

noteの有料記事500件の分析から導かれた、購入・クリックされるタイトルの設計指針。既存の「タイトル」セクション（疑問形 or 概念提示型）と組み合わせて使う。

### 購入されるタイトルの3要素

1. **具体性**: 「3ステップ」「〇〇の始め方」など、内容のイメージが一発で伝わる。抽象的な言葉だけで構成されたタイトルは弱い。
2. **読者のメリットが明確**: 「初心者でも実践できる」「作業効率が10倍になる」など、読み手にとっての価値が明示されている。
3. **トレンド性**: 生成AI、副業、SNSなど、今関心の高いテーマを自然に含んでいる。

### タイトル設計の4つのポイント

タイトルを考える際は、以下の4つの問いに答える形で設計する。

1. **誰に向けた内容か？**（例：副業初心者向け、デザイナー向け）
2. **どんな価値があるか？**（例：3ヶ月で初収益、設計力が上がる）
3. **どんな内容か？**（例：文章術の基本、AI活用の実践録）
4. **どう伝えるか？**（例：体験談、3ステップで解説、Before/After）

### 避けるべきパターンと改善例

| 避けるべき | 理由 | 改善例 |
|---|---|---|
| 「最近思ったこと」 | 内容が曖昧 | 「1年間AIを使い倒して見えた、デザイナーの生存戦略」 |
| 「自分なりの考え方」 | 対象が不明確 | 「BizDev3年目で気づいた、事業を伸ばす人の共通点」 |
| 「ブログを書いてみた」 | 一般的すぎる | 「はじめてのnote有料記事で100部売れた設計の裏側」 |
| 「やってみた記録」 | メリットが不明 | 「Claude Codeで業務を自動化した結果、週10時間が浮いた話」 |

### 國光スタイルとの統合

既存のタイトルスタイル（疑問形 or 概念提示型）に上記の3要素を組み合わせる：

- **疑問形 × 具体性**: 「AIネイティブな時代にデザイナーは何を100Xして生き残るのか？」← 數字（100X）＋ターゲット（デザイナー）＋トレンド（AI）が入っている
- **概念提示型 × メリット**: 「vibe designは、デザイナーの仕事をどう変えるのか」← トレンド（vibe design）＋ターゲット（デザイナー）＋価値（仕事が変わる）

タイトル案を出す際は、最低3案を出し、上記3要素の充足度で評価・選定すること。
