# Project-to-Visual

プロジェクトドキュメントを、美しいインタラクティブなシングルページHTML可視化に変換する。出力はGoogle Fonts以外に依存関係のない単一の自己完結型HTMLファイルで、スクロールベースのモジュール、アニメーション付きフロー図、アーキテクチャマップ、意思決定タイムライン（ADR形式）を通じてプロジェクト/システムの仕組みを教える。

**出自:** このスキルは Zara Zhang による [codebase-to-course](https://github.com/zarazhangrui/codebase-to-course) を改変したもの。元スキルはコードベースをVibe Coder向けのインタラクティブコースに変換する。本バージョンはプロジェクトドキュメントを、チーム横断のオンボーディング・認識合わせ用のインタラクティブ可視化に変換する。

## 初回ウェルカム

スキルが初めてトリガーされ、ユーザーがまだドキュメントを指定していない場合、自己紹介する：

> **プロジェクトのドキュメントを、インタラクティブな可視化ページに変換します。**
>
> 以下を指定してください：
>
> * **テキストファイル群** — 設計書、議事録、ADR、仕様書など
> * **フォルダ** — ドキュメントが入ったディレクトリ
> * **プロジェクト名と概要** — 何のプロジェクトか一言あると精度が上がります
>
> ドキュメントを読み込んで、アーキテクチャ全体像・データフロー・意思決定の経緯・進捗状況を、アニメーション付きのシングルHTMLファイルとして生成します。ブラウザだけで動作し、セットアップ不要です。

ユーザーがディレクトリパスを指定した場合は、`.md`、`.txt`、`.adoc`、`.rst`等のテキストファイルをすべてスキャンする。個別ファイルを指定した場合は、分析開始前にすべて読み込む。

## 対象ユーザー

想定読者は**同じ会社の別チームのメンバー** — 一般的な技術リテラシーはあるが、このプロジェクト固有のコンテキストを持っていない人。具体的には：

* プロジェクト途中から参加する新規アサインのエンジニア
* インターフェースやデータフローを理解する必要がある依存チームのメンバー
* 30のドキュメントを読まずにアーキテクチャ判断をレビューしたいステークホルダー
* プロジェクト開始から数ヶ月後にオンボーディングする将来のチームメンバー

**ドメイン知識はある前提、プロジェクトコンテキストはない前提。** API、データベース、マイクロサービスが何かは知っている — だが*なぜこのプロジェクトがPostgreSQLではなくDynamoDBを選んだのか*、*どのサービスが注文ライフサイクルを管理しているのか*は知らない。トーンは同僚同士：ホワイトボードの前でプロジェクトを案内する同僚であり、教壇から講義する教授ではない。

**読者の目的は実践的：**

* **アーキテクチャを素早く理解**して貢献・連携できるようにする — 「どんなコンポーネントがあり、誰が何を所有し、どう連携しているか？」
* **データフロー・業務フローをトレース**する — 「ユーザーがXをしたとき、エンドツーエンドで何が起こるか？」
* **なぜその判断をしたかを理解**する — 「なぜこのアプローチを選んだ？代替案は？何の制約がそれを駆動した？」
* **現在の状況を把握**する — 「何が完了し、何が進行中で、何がブロックされているか？」
* **正確にコミュニケーション**する — プロジェクトの語彙を習得し、曖昧さなく議論できるようにする

**全ドキュメントを読もうとしているわけではない。** 2日ではなく20分で、コード貢献・システム連携・フィードバック・意思決定ができる程度にプロジェクトを理解したい。

## このアプローチが有効な理由

従来のプロジェクトオンボーディング：README読む → 設計書15本流し読み → ミーティング3回出席 → それでも迷子。このアプローチはそれを反転させる：**まず全体像を見る → 気になる部分にズームイン → 各判断の「なぜ」を理解する。**

シングルファイル制約は意図的：HTMLファイル1つ＝セットアップ不要、Slack/メールで即共有、オフライン動作、設計判断の緊密化を強制。新メンバーが開いて1回で生産的になれる。

## プロセス（4フェーズ）

### フェーズ1：ドキュメント分析

HTMLを書く前に、プロジェクトを深く理解する。提供されたドキュメントをすべて読み、メンタルモデルを抽出し、プロジェクトの「ストーリー」を再構築する。

**抽出すべきもの：**

* **システムコンポーネント** — サービス、データベース、キュー、外部API、UIレイヤー。「登場人物」は誰か？
* **データフロー** — コンポーネント間でどんなデータが流れるか？各フローのトリガーは？正常パスとエラーパスは？
* **業務フロー** — システムがサポートするビジネスプロセスは？ユーザージャーニーのエンドツーエンドは？
* **アーキテクチャ判断（ADR）** — 何が、いつ、誰によって、なぜ決定され、どんな代替案が却下されたか？
* **制約とトレードオフ** — どんな非機能要件（レイテンシ、コスト、コンプライアンス、スケール）が設計を駆動したか？
* **現在の状態** — 何が完了し、何が進行中で、何が計画済みで、何がブロック/延期されているか？
* **主要インターフェース** — API契約、イベントスキーマ、チーム間の共有データモデル
* **プロジェクト固有の語彙** — このプロジェクト独自の用語、略語、命名規則

**ナラティブは自分で再構築する。** ユーザーに説明を求めない — 渡されたものがすべてかもしれない。可視化はプロジェクト/システムが何をするかを一文で説明することから始め、段階的に詳細を明らかにしていく。

### フェーズ2：モジュール設計

可視化を5〜8モジュールで構成する。アークは常に全体像（このシステムは何をするか？）から始まり、段階的にズームインする。カメラに例えると：衛星写真から始めて、都市、地区、建物へとズーム。

| モジュール | 目的 | 新メンバーにとっての意味 |
| --- | --- | --- |
| 1 | 「このシステムが何をするか — 30秒版」 | すべての土台。一文のミッション、主要ユーザージャーニー、ビジネスコンテキスト。 |
| 2 | コンポーネント紹介 | アーキテクチャマップ — 何があるか、誰が何を所有し、何と何が通信するか。「登場人物紹介」。 |
| 3 | データの流れ方 | 主要データ/業務フローのエンドツーエンドのアニメーショントレース。「ユーザーがXしたとき、こういう旅をする」。 |
| 4 | 副次フローとエッジケース | エラーハンドリング、リトライロジック、非同期処理、バッチジョブ — 知らないと壊す、目に見えにくいフロー。 |
| 5 | なぜこう作ったか（ADRタイムライン） | 意思決定タイムライン — 何が、いつ、なぜ決定され、何が却下されたか。「組織記憶」モジュール。 |
| 6 | インターフェースと契約 | API仕様、イベントスキーマ、共有データモデル — 連携するために他チームが知るべきこと。 |
| 7 | 現在の状態とロードマップ | 何が完了し、何が進行中で、何が次で、何がブロックされているか。スナップショット。 |

すべてのプロジェクトに7モジュール全部は不要。小さな社内ツールなら4〜5モジュールで十分。複雑な分散システムなら8モジュール必要かもしれない。プロジェクトの複雑さに合わせる。

**核心原則：** すべてのモジュールは読者が**行動**する助けになるべき — 貢献、連携、レビュー、意思決定。行動の助けにならないモジュールは削除するかリフレーミングする。

**各モジュールに含めるもの：**

* 3〜6画面（モジュール内を流れるサブセクション）
* 最低1つのアニメーション付きまたはインタラクティブなビジュアライゼーション（アーキテクチャ図、フローアニメーション、タイムライン等）
* 最低1つの「aha!」コールアウトボックス（重要な洞察やハマりポイント）
* 理解度チェック（クイズ）は含めない
* アーキテクチャを日常生活に結びつけるメタファー — ただしモジュール間で同じメタファーの使い回しは禁止
* プロジェクト固有の語彙ツールチップ（初出時）

**モジュール計画を承認に出さない — そのまま構築する。** ユーザーが欲しいのは可視化であり、計画書ではない。

### フェーズ3：可視化の構築

CSSとJavaScriptを埋め込んだ単一のHTMLファイルを生成する。`references/design-system.md`でCSSデザイントークン、タイポグラフィ、カラーシステムの全仕様を確認する。`references/interactive-elements.md`で全インタラクティブ要素の実装パターンを確認する。

**構築順序（タスクごと）：**

1. **土台を先に** — 全モジュールセクション（空）を持つHTMLシェル、完全なCSSデザインシステム、進捗トラッキング付きナビゲーションバー、スクロールスナップ動作、キーボードナビゲーション、スクロールトリガーアニメーション。このステップ後、スクロールできる動くスケルトンが完成しているべき。
2. **1モジュールずつ** — 各モジュールのコンテンツ、ダイアグラム、インタラクティブ要素を埋めていく。全モジュールを1パスで書こうとしない — 品質が落ちる。モジュール1を構築して動作確認、次にモジュール2、の順。
3. **仕上げパス** — 全モジュール構築後、トランジション、モバイルレスポンシブ、ビジュアル一貫性の最終パス。

**必須実装ルール：**

* ファイルは完全に自己完結（外部依存はGoogle Fonts CDNのみ）
* CSSは `scroll-snap-type: y proximity` を使用（`mandatory`は禁止 — mandatoryは長いモジュール内でユーザーを閉じ込める）
* セクションには `min-height: 100dvh`（フォールバックとして `100vh`）
* アニメーションは `transform` と `opacity` のみ（GPU性能のため）
* 全JSをIIFEでラップ、スクロールリスナーに `passive: true`、`requestAnimationFrame` でスロットル
* タッチ対応、キーボードナビゲーション（矢印キー）、ARIA属性を含める
* 日本語・英語両方のコンテンツを自然にサポート — テキストレンダリング、改行挙動、フォントフォールバック

### フェーズ4：レビューとオープン

生成したHTMLファイルは `output/visual/` ディレクトリに保存する。

```bash
mkdir -p output/visual
# 例: output/visual/<project-name>.html
```

ブラウザで開いてユーザーにレビューしてもらう。構築したものをウォークスルーし、コンテンツ、デザイン、インタラクティビティについてフィードバックを求める。

## コンテンツ哲学

この原則が、優れた可視化と凡庸なダイアグラム集を分ける。

### 見せる、語るな — 徹底的にビジュアル

プロジェクトドキュメントはすでにテキスト。このスキルの存在意義はテキストをビジュアルに変換すること。以下のルールを厳守：

**テキスト制限：**

* テキストブロックは最大**2〜3文**。4文目を書きそうになったら、ビジュアルに変換する。
* すべての画面は**最低50%がビジュアル**（ダイアグラム、フローチャート、カード、アニメーション、タイムライン — 段落以外のもの）。

**テキストをビジュアルに変換：**

* コンポーネント一覧 → **アイコンと接続線付きのアーキテクチャ図**
* ステップの連続 → **データがノード間を移動するアニメーション付きフロー図**
* 「サービスAがサービスBを呼び、それがDB Cに書き込む」 → パスをハイライトした**アニメーション付きデータフロー**
* 意思決定の時系列リスト → 展開可能なカード付き**インタラクティブタイムライン**
* アプローチの比較 → ✓/✗インジケーター付き**サイドバイサイドカラム**
* 作業項目のステータス → **進捗カードまたはカンバン風ボード**
* API契約 → シンタックスハイライト付き**リクエスト/レスポンスペアカード**

### アーキテクチャ図 — ヒーロービジュアル

アーキテクチャ図は出力全体で最も重要なビジュアル。以下を満たすこと：

* すべての主要コンポーネントを、アイコン付きの識別可能なノードとして表示（🗄️ データベース、⚡ ファンクション、🌐 API、📨 キュー等）
* コンポーネント間の流れをラベル付き矢印で表示（データタイプ、プロトコル）
* レイヤーを色分け（フロントエンド / バックエンド / データ / 外部）
* インタラクティブ：コンポーネントのクリック/ホバーで接続をハイライトし、責務のツールチップを表示
* モジュール2に配置し、全体で参照/再利用

### フローアニメーション — 旅をたどる

主要フロー（ユーザーアクション → システム応答）ごとに、以下のアニメーションを作成：

* データが通過する順にコンポーネントを順次点灯
* 各ステップでデータがどう変換されるかを表示（入力 → 出力）
* 再生/一時停止/ステップ制御あり
* アニメーションと並行した「平易な言葉」のナレーショントラック（「まずリクエストがAPIゲートウェイに到達し、認証トークンを検証…」）

### 意思決定タイムライン（ADR形式）

アーキテクチャ判断のインタラクティブタイムライン：

* 判断を水平または垂直軸上に時系列で表示
* 各判断ノードを展開すると表示：コンテキスト、判断内容、検討した代替案、結果
* 判断ステータスを色分け：✅ 採択、⚠️ 廃止、🔄 レビュー中
* 関連する判断をリンク（例：「この判断は判断#3に起因する」）

### テクニカル ↔ ビジネス翻訳

すべての技術的概念に、ビジネスインパクトへのサイドバイサイド翻訳を付ける。左パネル：技術的記述。右パネル：ビジネス/ユーザーにとっての意味。

例：
* 「結果整合性を使用」 → 「データのリージョン間同期に数秒かかる可能性 — ユーザーは一瞬古いデータを見る場合がある」
* 「決済サービスにサーキットブレーカーパターン」 → 「決済が壊れても、アプリの他の部分は全面クラッシュせず動き続ける」
* 「PostgreSQLではなくDynamoDBを選択」 → 「複雑なクエリよりスピードとスケールを優先 — レポーティングには別ソリューションが必要」

### 語彙ツールチップ — プロジェクト固有

すべてのプロジェクトに独自の専門用語がある — 内部サービス名、略語、ドメイン用語。積極的にツールチップを付ける。

* 内部サービス名（「OrderOrchestrator」「PaymentGateway」）— 何をするかをツールチップ
* ドメイン略語（「PII」「SLA」「CQRS」）— 定義とプロジェクト固有のコンテキストをツールチップ
* プロジェクト固有用語（「ハイドレーション」「バックフィル」「レガシーパス」）— *このプロジェクトでの*意味をツールチップ

**カーソル：** 用語には `cursor: pointer` を使用。**ツールチップ位置：** `position: fixed` と `getBoundingClientRect()` を使用し、`document.body` に追加して `overflow: hidden` コンテナによるクリッピングを回避。

## デザインアイデンティティ

ビジュアルデザインは**モダンな社内ダッシュボード** — クリーンでプロフェッショナル、Algomaticブランドカラーで統一。`reference/design-system.md` でトークンシステムの全仕様を確認。

譲れない原則：

* **ブランドカラーパレット**：白背景（#FFFFFF）、ライトブルー（#CFE3F3）、ライトグレー（#D9D9D9）。ダークネイビー（#1B394F）とミディアムブルー（#386F9E）をアクセントに。暖色系やティール、コーラルは使わない。
* **ブルー基調のアクセント**：ダークネイビー（#1B394F）をプライマリアクセント、ミディアムブルー（#386F9E）をセカンダリに。パープルグラデーションは禁止。
* **個性的なタイポグラフィ**：見出しに個性のあるディスプレイフォント（Bricolage Grotesqueまたは同等）。本文にクリーンなサンセリフ（DM Sans）。技術用語にJetBrains Mono。
* **ゆとりある余白**：モジュールは呼吸する。画面あたり最大3〜4の短い段落。
* **交互背景**：偶数/奇数モジュールで白（#FFFFFF）とライトブルー（#CFE3F3）を交互に
* **コンポーネントノード**：各システムコンポーネントに一貫したビジュアルアイデンティティ（アイコン＋色）を全体で使用。ブランドカラーの濃淡で識別。
* **ネイビー基調のシャドウ**：rgba(27, 57, 79, 0.08〜0.14) の微かなシャドウ。黒いドロップシャドウは禁止。

## ハマりどころ — よくある失敗

### ツールチップのクリッピング

`overflow: hidden` を持つコンテナがツールチップを切り取る。**対策：** ツールチップは `position: fixed` で `document.body` に追加。`getBoundingClientRect()` で位置を計算。

### アーキテクチャ図が密すぎる

すべてのマイクロサービスとデータベースを1つの図に詰め込もうとする。**対策：** プログレッシブディスクロージャーを使用 — トップレベルの5〜7コンポーネントから始め、クリックでサブシステムを展開。

### テキストの壁

可視化がスタイリングが良くなっただけの元ドキュメントに見える。**対策：** すべての画面は最低50%がビジュアル。2〜3文以上ビジュアルブレイクなしで書いていたら、止める。

### フローアニメーションの欠如

静的ダイアグラムしかないモジュール。**対策：** すべての主要フローにアニメーションが必要 — 動きこそが、ものごとの繋がりを教える。

### コンテキストのないADR

制約の説明なしに判断を列挙する。**対策：** すべての判断カードに含める：「Yの理由でXが必要だったので、Wではなく、Zを選んだ」。

### Scroll-Snap Mandatory

`scroll-snap-type: y mandatory` は長いモジュール内にユーザーを閉じ込める。常に `proximity` を使用。

### モジュール品質の劣化

全モジュールを1パスで書こうとする。**対策：** 1モジュールずつ構築し、次に進む前に各モジュールを検証。

### インタラクティブ要素の欠如
