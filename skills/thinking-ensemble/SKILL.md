# アンサンブル思考

> **核心**: 問題定義 → タスク分類 → 4グループ並列起動 → 重みづけ統合

16の性格タイプを4グループ×4エージェントの「認知チーム」として扱い、**常に全4グループを並列サブエージェント** で起動する。
各グループ内では4つの認知レンズが動的に対話し（Multi-Agent Systems）、グループ固有の結論を導出する。

並列実行により各グループは独立したコンテキストで動作するため、グループ数を絞る必要がない。
Task ツールの並列上限4と4グループが一致し、認知的多様性を最大化しながら処理速度も最大化する。

```
C = A(NT, NF, SJ, SP)

NT/NF/SJ/SP: 4エージェントが対話する認知チーム（常に全4グループを並列実行）
A():          タスク分類に基づく重みづけ統合（オーケストレーター）
目標:         認知的多様性の最大化 × 処理速度の最大化
```

## 4グループ一覧

| グループ | エージェント構成 | チーム特性 | 参照 |
|----------|-----------------|-----------|------|
| **NT: 論理的革新** | INTJ(設計者) + INTP(分析者) + ENTP(挑戦者) + ENTJ(決断者) | 戦略×理論×仮説×実行 | [analysts-nt.md](analysts-nt.md) |
| **NF: 共感的意味付け** | INFJ(洞察者) + INFP(価値評価者) + ENFP(接続者) + ENFJ(統合者) | 理想×価値×創発×導き | [diplomats-nf.md](diplomats-nf.md) |
| **SJ: 堅実な管理** | ISTJ(検証者) + ESTJ(組織者) + ISFJ(守護者) + ESFJ(調和者) | 検証×統制×保護×協調 | [sentinels-sj.md](sentinels-sj.md) |
| **SP: 実践的適応** | ISTP(実験者) + ESTP(行動者) + ISFP(感覚者) + ESFP(観察者) | 実験×即断×審美×体験 | [explorers-sp.md](explorers-sp.md) |

## タスク別統合重みづけガイド

全4グループを常に起動するが、統合時にタスク特性に応じた重みづけを行う。
重みの高いグループの結論を軸に、低いグループからは「予想外の洞察」を拾う。

| タスク | 重み高 | 重み中 | 補完的洞察 |
|--------|-------|-------|-----------|
| 設計・アーキテクチャ | NT | SJ | NF, SP |
| バグ調査 | NT, SP | — | NF, SJ |
| UX/UI設計 | NF, SP | — | NT, SJ |
| コードレビュー | SJ, NT | — | NF, SP |
| チーム意思決定 | NT, NF | SJ | SP |
| リファクタリング | NT, SP | — | NF, SJ |
| リスク評価 | SJ | NT | NF, SP |
| 新機能企画 | NF, NT | SP | SJ |
| オンボーディング | NF, SJ | — | NT, SP |
| パフォーマンス最適化 | NT, SP | — | NF, SJ |

**マップにないタスク**: 以下の軸で重みを判断

```
分析・戦略が核心 → NT を重み高に
共感・意味付けが核心 → NF を重み高に
検証・管理が核心 → SJ を重み高に
実験・行動が核心 → SP を重み高に
```

## 実行プロトコル

```
1. [問題定義]
   解くべき問いを1文で定義する
   例: 「認証フローのリファクタリング方針を決定する」

2. [タスク分類]
   上記ガイドからタスクタイプを特定し、統合時の重みづけを確認する
   → マップにない場合は4軸で重みを判断

3. [4グループ並列起動]
   4つのグループファイルを Read ツールで読み込む
   → 「サブエージェントプロンプトテンプレート」に従い Task ツールを構成
   → 4つのサブエージェントを同一メッセージ内で起動（並列実行）

4. [重みづけ統合]
   4a. 各グループの結論を収集する
   4b. 重みづけ: タスク分類に基づき、各グループの結論に重みを設定
   4c. 収束点: 複数グループが同じ方向を指している点を特定
   4d. 発散点: グループ間で矛盾・対立している点を特定
   4e. 補完的洞察: 重みの低いグループから予想外の視点を拾う
   4f. 結論: 重み高グループを軸に、発散点を条件付きで統合
```

## サブエージェントプロンプトテンプレート

Step 3 で4グループのサブエージェントを起動する際に使用する。
**事前に4つのグループファイルを Read ツールで読み込み、各 `{グループファイル全文}` に展開すること。**

### Task ツール設定

```
subagent_type: "thinking-analyst"
model: "fast"
description: "{グループ略称}グループ対話"
```

### プロンプト本文

```
あなたは「{グループ名}」チームの4つの認知エージェントとして振る舞います。
各エージェントの認知レンズに基づいた対話サイクルを実行し、グループとしての結論を導出してください。

## 問題定義
{問題定義（Step 1 で定義した1文）}

## タスクタイプ
{タスクタイプ（Step 2 で分類した結果）}

## 認知レンズ定義と対話プロトコル
{グループファイル全文}

## 実行指示
1. 問題定義を各認知レンズの「問いの生成パターン」に適用し、各レンズ固有の問いを生成する
2. 「グループ内マルチエージェント協調プロトコル」の対話サイクルに従い、4エージェントの対話を実行する
3. 対話ルールの収束条件・発散条件に従い、必要に応じて追加サイクルを実行する（最大3サイクル）
4. 「グループ出力の形式」に従い、対話ログとグループ結論を出力する

## 制約
- 各エージェントは自身のコアレンズと認知的強みに忠実に発話すること
- 対話は建設的に行い、単なる同意ではなく各レンズ固有の視点を提供すること
- グループ結論は全エージェントの洞察を統合したものであること
```

## 簡易モード（サブエージェント不使用）

軽微なタスクや即座に結論が欲しい場合、サブエージェントを起動せず逐次実行も可能。

```
判断基準:
- 問題が単純で、各グループの問いを列挙するだけで十分 → 簡易モード
- 深い分析が必要 → 並列サブエージェントモード（デフォルト）
```

簡易モードでは、4グループの「問いの生成パターン」のみを適用し、対話サイクルは省略する。

## 既存思考スキルとの併用

アンサンブル思考は「どの視点で見るか」を決める。既存スキルは「どう推論するか」を決める。

| フェーズ | ensemble の役割 | 併用する既存スキル |
|----------|----------------|-------------------|
| 問題定義 | — | メタ思考（問い自体の妥当性） |
| 並列分析 | 4グループで同時に対話を実行 | ロジカルシンキング（問いの構造化） |
| 統合 | 重みづけ・収束/発散/盲点の整理 | ディベート思考（対立点の解消） |
| 検証 | — | クリティカルシンキング（結論の検証） |

**典型的な組み合わせ**:

- 設計判断: ensemble(全4グループ) → thinking-debate(グループ間対立の解消) → thinking-logical(結論構造化)
- バグ調査: ensemble(全4グループ) → thinking-abduction(仮説形成) → thinking-critical(検証)
- リスク評価: ensemble(全4グループ) → thinking-mece(網羅分解) → thinking-pac(前提検証)

## アンチパターン

| やりがち | 問題 | 対策 |
|---------|------|------|
| 簡易モードで済む問題にサブエージェント起動 | オーバーヘッドで非効率 | 判断基準を確認し適切なモードを選択 |
| 全グループの結論を平等に扱う | 重要度の差を無視 | タスク分類に基づき重みづけして統合 |
| 重みの低いグループの結論を無視する | 予想外の洞察を逃す | 補完的洞察として必ず拾う |
| 性格タイプに感情移入 | 分析が主観的に歪む | レンズは「問いの生成器」であり人格ではない |
| 既存思考スキルと混同 | 視点と推論手法が混線 | ensemble=視点、既存=手法と使い分ける |
| 統合フェーズを省略 | 並列実行した意味が失われる | 収束点・発散点・盲点を必ず明示 |

## 注意

**レンズは「問いの生成器」であり、人格シミュレーションではない**

各レンズが提供するのは「その認知特性を持つ人なら、ここで何を問うか」という問いの視点。
レンズを通じて問いを立て、その問いに対する答えを自分の知識と論理で導く。
グループ内の対話サイクルは、異なる認知特性の相互作用で盲点を補完する仕組みである。
