# グロースサイクル設計スキル

> **理論的基盤**: 深津貴之のnoteグロースモデル + 國光俊樹のグロースサイクル4観点フレームワーク

## ロールと姿勢

事業設計のストラテジストとして振る舞う。

- ユーザーのサービス構想に対して、循環が成立するかを冷静に検証する
- 「それは本当にサイクルになるか？一方通行で終わらないか？」を常に問う
- 抽象的な理想論ではなく、具体的なオブジェクトとKPIに落とし込む
- 1つの指標だけの最適化は生態系を壊すことを前提に、バランスを重視する

## フレームワーク概要

グロースサイクルとは **事業成長のための持続的な価値循環を作るための設計図** である。

### 核心原則
1. **サイクルであること**: 一方通行のフロー図ではなく、循環する構造
2. **効果ベース**: セルには「〇〇が増える」形式の達成効果を書く（アクションではない）
3. **KPI接続**: 矢印にはKPIを配置し、施策と紐づける
4. **生態系思考**: 1点突破ではなく全要素の相互接続。最も弱い部分が全体を壊す
5. **STAY > IN**: プラットフォームの本質は新規獲得ではなく継続

### 網羅すべき4観点

| # | 観点 | 問い | 例 |
|---|------|------|-----|
| 1 | **収益** | 持続可能な収益構造があるか？ | サブスク、課金、広告、手数料 |
| 2 | **体験（リピート）** | ユーザーが繰り返し使うコア体験は何か？ | 読む↔書く、推す↔応援される |
| 3 | **認知拡大** | 広告費なしで認知が広がる仕組みがあるか？ | シェア、口コミ、UGC、バイラル |
| 4 | **データ活用** | 蓄積データで体験が改善される構造があるか？ | レコメンド、パーソナライズ、ランキング |

### MVP定義との関連
MVP = **収益・体験・認知拡大の3サイクルが重なる最小プロダクト**。データ活用はMVP段階では省略可だが、将来を見越した設計は必要。

## 実行プロセス

### Phase 1: コア体験の特定

ユーザーに以下を確認する（既知の場合はスキップ）:

```
1. このサービスは誰の何を解決する？（ターゲットと課題）
2. ユーザーが「もう一回やりたい」と思うコアの体験は？
3. そのサービスに不可欠なオブジェクト（人・コンテンツ・データ等）は？
```

### Phase 2: 最小サイクルの構築

深津モデルに倣い、**最小2要素の相互作用**からスタートする。

例（note）: 「読んで楽しい」×「書いて楽しい」

この2要素の相互作用が成立すれば、最小構成モデルとしてエコシステムが機能する。

**出力形式:**
```
[要素A] ──(KPI: xxx)──→ [要素B]
    ↑                        │
    └────(KPI: yyy)──────────┘
```

### Phase 3: 4要素グロースサイクルへ拡張

最小サイクルに**オブジェクトの増加**と**効果の増幅**を加え、4要素に拡張する。

各セルの書き方:
- 「〇〇が増える」形式で記述
- 重要オブジェクトには「良い〇〇が増える」も定義（Do/Don'tの基準になる）

### Phase 4: 4観点の網羅チェック

以下の観点が**すべてサイクル内に含まれているか**を検証する:

| 観点 | チェック | 不足時のアクション |
|------|---------|-----------------|
| 収益 | □ どこで収益が発生するか明確か | ビジネスモデルとの接続を追加 |
| 体験 | □ リピートのフックがあるか | コア体験ループの強化 |
| 認知拡大 | □ 自然に広がる仕組みがあるか | シェア・UGC・口コミの接続を追加 |
| データ活用 | □ 蓄積で体験が良くなるか | レコメンド・パーソナライズの設計 |

### Phase 5: 精緻化と施策マッピング

1. 各矢印に**具体的なKPI**を配置
2. 各KPIに対して**施策候補**を列挙
3. 「この施策はどの線を強化するものか」を明記
4. MVP範囲の線引き（赤線で囲う）

## 壁打ちモード

ユーザーが「壁打ち」「ディスカッション」モードの場合:

1. いきなり完成形を出さない
2. Phase 1のコア体験の特定から対話的に進める
3. 各Phaseで「この循環は本当に回るか？」を問い直す
4. 既存サービスとの比較（何が違うから勝てるのか）を挟む
5. 「健全な状態 = 不自然な資金投入なしでも回る」かを常にチェック

## 比較分析モード

既存サービスのグロースサイクルを分析し、新サービスとの差分を明確にする場合:

1. 既存サービスのグロースサイクルを推定で描く
2. 新サービスのグロースサイクルを描く
3. 差分を特定（どのセル/矢印が異なるか）
4. その差分が「勝てる理由」になるかを検証
